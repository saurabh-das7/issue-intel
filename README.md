# llm-issue-categorizer

A practical, PM-driven tool for turning noisy operational issue data into structured, 
actionable product intelligence — built from real experience managing vendor operations 
where ticket labels are vague, inconsistent, and useless for pattern detection.

This repo documents both the **thinking** (frameworks, PRDs, risk registers) and the 
**doing** (a working tool you can run on your own data).

---

## Current Tool — Issue Categorisation & Pattern Detection

> **Your vendor team resolves hundreds of issues every week. Your ticket labels tell you nothing.**

The Issue Categorisation & Pattern Detection tool is a Streamlit web application that 
accepts a CSV, Excel, or TXT file of operational issues — tagged with whatever vague 
labels your team assigned — re-categorises every row against a structured taxonomy 
using an LLM, and surfaces the patterns a PM needs to make product decisions. 
Processes up to 100 rows in under 90 seconds.

### Three Ways to Categorise

**No suggestions** — the LLM reads all your rows and generates its own category names 
from scratch. No input required. Useful when you genuinely don't know what your 
categories should be before seeing the data.

**Manual list** — you define the starting categories, the LLM maps tickets to them 
and creates additional buckets only when the data warrants it. Useful when you already 
have a taxonomy in mind and want the LLM to fill it in.

**Auto-suggest** — the app samples 25% of your rows, proposes a taxonomy, you 
review and edit it, then the full run begins. The LLM can still create new categories 
during the run for patterns the sample missed. Available for datasets over 50 rows.

This mirrors how a real PM approaches the problem. You don't know your final categories 
until you've seen your data — but you might have a starting point worth keeping.

### Three Outputs From One Run

**Single-bucket view** — each ticket receives exactly one primary category. A 
scrollable results table with confidence scoring (High / Medium / Low) and a 
one-line reasoning note per row. A horizontal bar chart shows the distribution. 
Filter to Low confidence rows only for quick spot-checking before downloading.

**Advanced multi-tag view** — run on demand after the primary results. Each ticket 
is mapped to all applicable themes, not just the primary one. A second bar chart 
shows theme frequency across the full dataset — the same ticket can contribute to 
multiple bars. Answers a different question: not "what is the primary issue?" but 
"which themes appear most often, even as secondary signals?"

**Post-run review panel** — generated by a final LLM pass after categorisation 
completes. Contains four sections: a transparency log of every auto-merge applied 
(with the consolidated label and ticket count), a one-sentence summary per category 
in plain language, merge flags for uncertain cases the LLM did not auto-apply, and 
an analysis of the Uncategorised bucket surfacing themes that did not reach category 
threshold.

Every flow is zero-friction — a first-time user can experience the full output 
with two clicks using the built-in sample datasets.

### What Makes It Smart

**Context-aware categorisation.** You describe your data in one line before running 
("Support tickets for a logistics ops team"). That description is passed to the LLM 
with every prompt. "Slow response" means something different in a payments app versus 
an AI assistant — the tool uses context to make that distinction.

**Auditability over automation.** Every categorisation decision includes a one-line 
reasoning note. The post-run review panel extends this to the category level. 
High-confidence merges are auto-applied but logged transparently — the user can see 
exactly what was consolidated and why.

**Thresholds create signal, not noise.** Categories below the minimum threshold 
(5 rows) are collapsed into an Uncategorised bucket and analysed separately. 
A chart with 6–8 meaningful categories plus an analysed Uncategorised bucket is 
more useful to a PM than one with 20 categories where 14 have 1–2 rows each.

## Why This Exists

Vendor teams and support operations resolve hundreds of issues every week. But the 
labels they assign — "quality issue", "performance problem", "ad not serving" — are 
written for resolution speed, not for analysis. The result is a graveyard of ticket 
data that no one mines.

Product improvements that should be obvious stay invisible because the signal is 
buried in noise. A PM inheriting this data cannot tell whether they have a systematic 
infrastructure problem, a training gap in the vendor team, or a product bug driving 
the same failure mode repeatedly — because the labels do not distinguish between them.

This tool is the missing layer between raw operational data and product decisions: 
consistent categorisation at scale, with the reasoning surfaced alongside every label.

## How to Run This Locally

**Requirements:** Python 3.9+, a free Gemini API key from 
[aistudio.google.com](https://aistudio.google.com) (no credit card required)

```bash
# Clone the repo
git clone https://github.com/saurabh-das7/llm-issue-categorizer.git
cd llm-issue-categorizer

# Install dependencies
pip install -r requirements.txt

# Add your API key
export GOOGLE_API_KEY=your_key_here

# Run the app
streamlit run app/main.py
```

## Who This Is For

**If you manage vendor operations, support queues, or customer feedback pipelines** 
— this tool replaces manual re-labelling and pattern-spotting with a consistent, 
auditable, LLM-powered categorisation layer you can run on any CSV, Excel, or TXT export.

**If you're a PM or TPM building AI products** — the docs folder is a worked example 
of how to frame, spec, and ship an LLM-powered tool from scratch, including the 
thinking behind every product and technical decision.

**If you're exploring how to use LLMs for structured data classification** — the 
three-mode taxonomy design, confidence scoring, batch consistency architecture, and 
two-pass consolidation approach are directly transferable to any domain where you 
need LLM-powered labelling at scale.
